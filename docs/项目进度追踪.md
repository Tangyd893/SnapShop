## 五、风险与观察

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 自动化测试环境依赖 Nacos | CI 或本地一键跑测不可用 | 增加 test profile 独立配置，避免硬依赖外部配置中心 |
| C 端构建失败未被 CI 捕获 | 主干可能存在前端回归 | 将 `snapshop-web` 构建纳入强制校验，并修复别名解析 |
| MQ 消费异常路径未端到端验证 | 可能出现重复 nack、事务回滚与 Redis 结果状态不一致 | 用集成测试覆盖库存服务失败、订单创建失败、ack 失败模拟 |
| 消费日志状态只保留 `PROCESSING`/`SUCCESS` | 排障时失败原因可能依赖日志而非表记录 | 若需要运维查询，可增加失败原因记录或死信表联动 |
| 支付脚本依赖已有订单 | 新环境可能无法完整跑完支付链路 | 在脚本前置步骤中补充建单或秒杀成功准备数据 |
| 压测活动与 SKU 默认为 1 | 测试数据不一致会导致压测误判 | 压测前固定初始化活动、库存和用户数据 |
