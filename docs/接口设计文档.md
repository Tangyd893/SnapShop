# SnapShop 在线购物秒杀平台接口设计文档

## 一、文档说明

本文档用于定义 SnapShop 在线购物秒杀平台的前后端接口、服务间接口以及 RabbitMQ 消息契约。接口统一通过网关服务对前端暴露，前端使用 Vue3 和 Axios 调用接口。

接口设计目标：

- 前端调用路径清晰，便于 Vue3 页面开发。
- 后端接口职责边界明确，便于微服务拆分。
- 秒杀提交接口尽量轻量，避免同步长事务。
- 所有写操作具备请求编号，方便幂等控制和链路追踪。
- 秒杀异步下单链路具备明确的消息结构和状态返回。

## 二、接口通用规范

### 2.1 基础路径

所有前端接口统一通过网关访问：

```text
/api
```

服务路径规划：

| 路径前缀 | 目标服务 | 说明 |
| --- | --- | --- |
| `/api/auth` | 认证服务 | 登录、注册、令牌刷新 |
| `/api/users` | 用户服务 | 用户资料、收货地址 |
| `/api/products` | 商品服务 | 商品列表、商品详情 |
| `/api/seckill` | 秒杀服务 | 秒杀活动、秒杀令牌、秒杀提交、秒杀结果查询 |
| `/api/orders` | 订单服务 | 订单查询、订单取消 |
| `/api/payments` | 支付服务 | 支付单、模拟支付、支付结果 |

### 2.2 请求头规范

| 请求头 | 是否必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 登录后必填 | 登录令牌，格式为 `Bearer {令牌}` |
| `X-Request-Id` | 写操作必填 | 请求编号，前端生成或网关生成 |
| `X-Client-Type` | 建议填写 | 客户端类型，例如网页端 |
| `Content-Type` | 提交请求必填 | 固定为 `application/json` |

请求编号用于日志追踪和幂等控制。对于秒杀提交、下单、支付、取消订单等写操作，前端应保证一次用户操作对应一个请求编号。

### 2.3 统一响应格式

```json
{
  "code": 0,
  "message": "成功",
  "data": {},
  "requestId": "REQ202605270001",
  "timestamp": "2026-05-27 10:00:00"
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `code` | 业务状态码，`0` 表示成功 |
| `message` | 业务提示信息 |
| `data` | 响应数据 |
| `requestId` | 请求编号 |
| `timestamp` | 服务端响应时间 |

### 2.4 分页响应格式

```json
{
  "code": 0,
  "message": "成功",
  "data": {
    "records": [],
    "pageNo": 1,
    "pageSize": 10,
    "total": 100,
    "pages": 10
  },
  "requestId": "REQ202605270001",
  "timestamp": "2026-05-27 10:00:00"
}
```

### 2.5 通用状态码

| 状态码 | 含义 |
| --- | --- |
| `0` | 成功 |
| `40000` | 请求参数错误 |
| `40100` | 未登录或令牌无效 |
| `40300` | 无权限 |
| `40400` | 资源不存在 |
| `40900` | 请求冲突或重复提交 |
| `42900` | 请求过于频繁 |
| `50000` | 系统异常 |
| `50010` | 远程服务调用失败 |
| `50020` | 消息发送失败 |

### 2.6 秒杀业务状态码

| 状态码 | 含义 |
| --- | --- |
| `60000` | 活动未开始 |
| `60001` | 活动已结束 |
| `60002` | 商品已售罄 |
| `60003` | 用户已参与 |
| `60004` | 用户不满足参与资格 |
| `60005` | 秒杀请求排队中 |
| `60006` | 秒杀订单创建成功 |
| `60007` | 秒杀订单创建失败 |
| `60008` | 秒杀结果不存在或已过期 |

### 6.5 提交秒杀请求

```text
POST /api/seckill/activities/{activityId}/items/{skuId}/submit
```

请求头：

```text
Authorization: Bearer 访问令牌
X-Request-Id: REQ202605270016
```

请求体：

```json
{
  "seckillToken": "秒杀令牌",
  "quantity": 1
}
```

成功响应体：

```json
{
  "code": 60005,
  "message": "秒杀请求已提交，正在排队处理",
  "data": {
    "requestId": "REQ202605270016",
    "resultStatus": "排队中"
  },
  "requestId": "REQ202605270016",
  "timestamp": "2026-05-27 10:00:00"
}
```

失败响应体示例：

```json
{
  "code": 60002,
  "message": "商品已售罄",
  "data": {
    "resultStatus": "售罄"
  },
  "requestId": "REQ202605270016",
  "timestamp": "2026-05-27 10:00:00"
}
```

处理规则：

- 秒杀提交接口必须登录。
- 数量固定为 1 或不超过活动限购数量。
- 服务端根据用户编号、活动编号、商品规格编号判断是否重复参与。
- Redis 预扣库存成功后才投递 RabbitMQ。
- 接口只返回排队中，不同步返回订单编号。

### 6.6 查询秒杀结果

```text
GET /api/seckill/results/{requestId}
```

**职责说明**：

- 网关将请求路由到**秒杀服务**。
- **订单服务**在消费秒杀下单消息并完成业务后，负责**写入** Redis 键 `seckill:result:{requestId}`（及必要时的数据库快照）。
- **秒杀服务**对外提供查询接口：优先读 Redis；未命中时回查订单服务内部接口或数据库，组装 `resultStatus`、`orderId` 等字段返回。

响应体：

```json
{
  "code": 0,
  "message": "成功",
  "data": {
    "requestId": "REQ202605270016",
    "resultStatus": "成功",
    "orderId": 50001,
    "orderNo": "SO202605270001",
    "failureReason": null
  },
  "requestId": "REQ202605270017",
  "timestamp": "2026-05-27 10:00:00"
}
```

结果状态：

| 状态 | 说明 |
| --- | --- |
| `排队中` | 消息已提交，订单尚未创建 |
| `成功` | 订单创建成功 |
| `失败` | 订单创建失败 |
| `售罄` | 库存不足 |
| `重复参与` | 用户已经参与过 |
| `结果不存在` | 请求编号不存在或结果已过期 |

**前端轮询契约**（完整增强版必须遵守）：

| 参数 | 固定值 |
| --- | --- |
| 轮询间隔 | 2000 毫秒 |
| 最大轮询次数 | 15 次 |
| 超时后提示 | “处理中，请稍后在订单列表查看” |

超过最大次数且仍为 `排队中` 时，前端停止轮询并展示上述提示，用户可从订单列表继续查看。

```json
  "requestId": "REQ202605270023",
  "timestamp": "2026-05-27 10:00:00"
}
```

## 九、服务间接口

服务间接口通过 OpenFeign 调用，不直接暴露给前端。写操作需要传递请求编号或业务幂等键。

### 9.1 用户服务校验用户状态

```text
GET /internal/users/{userId}/status
```

响应体：

```json
{
  "userId": 10001,
  "status": "正常",
  "allowSeckill": true
}
```

### 9.2 商品服务查询秒杀商品信息

```text
GET /internal/products/skus/{skuId}/seckill-info
```

响应体：

```json
{
  "skuId": 30001,
  "productId": 20001,
  "title": "秒杀测试商品",
  "originPrice": 19900,
  "status": "上架"
}
```

### 9.3 库存服务扣减库存

```text
POST /internal/inventory/deduct
```

请求体：

```json
{
  "requestId": "REQ202605270016",
  "businessKey": "10001:10001:30001",
  "skuId": 30001,
  "quantity": 1,
  "scene": "秒杀下单"
}
```

响应体：

```json
{
  "success": true,
  "failureReason": null
}
```

处理规则：

- 必须使用条件更新防止库存扣成负数。
- 必须记录库存流水。
- 同一个业务键重复请求不能重复扣减。

### 9.4 库存服务回补库存


请求体：

```json
{

  "quantity": 1,
  "reason": "订单超时关闭"
}
```

响应体：


### 9.5 订单服务写入秒杀结果（内部）

订单服务消费秒杀消息后调用，或由订单服务内部直接写入 Redis（推荐在订单服务内封装，不对外暴露 HTTP）。

Redis 键：`seckill:result:{requestId}`

值示例：

```json
{
  "requestId": "REQ202605270016",
  "resultStatus": "成功",
  "orderId": 50001,
  "orderNo": "SO202605270001",
  "failureReason": null
}
```

| 字段 | 说明 |
| --- | --- |
| TTL | 86400 秒（24 小时） |
| 写入时机 | 订单创建成功、失败、售罄、重复参与等业务终态时 |

秒杀服务查询接口（§6.6）优先读取该键；未命中时可调用订单服务：

```text
GET /internal/orders/seckill-result/{requestId}
```

## 十、RabbitMQ 消息契约

### 10.1 秒杀下单消息

交换机：

```text
seckill.order.exchange
```

队列：

```text
seckill.order.queue
```

路由键：

```text
seckill.order.create
```

消息体：

```json
{
  "messageId": "秒杀消息:10001:30001:10001:REQ202605270016",
  "requestId": "REQ202605270016",
  "businessKey": "10001:10001:30001",
  "userId": 10001,
  "activityId": 10001,
  "skuId": 30001,
  "quantity": 1,
  "seckillPrice": 9900,
  "createdAt": "2026-05-27 10:00:00"
}
```

消费规则：

- 订单服务消费该消息。
- 根据 `messageId` 做消息幂等。
- 根据 `businessKey` 做业务幂等。
- 订单创建成功后手动确认消息。
- 订单创建失败时根据异常类型决定重试或进入死信队列。

### 10.2 订单超时消息

交换机：

```text
order.timeout.exchange
```

队列：

```text
order.timeout.queue
```

消息体：

```json
{
  "messageId": "订单超时:SO202605270001",
  "orderId": 50001,
  "orderNo": "SO202605270001",
  "userId": 10001,
  "expireAt": "2026-05-27 10:15:00"
}
```

消费规则：

- 订单仍为待支付时关闭订单。
- 订单已支付时忽略消息。
- 关闭订单后触发库存回补。

### 10.3 支付成功消息

交换机：

```text
payment.event.exchange
```

队列：

```text
payment.success.queue
```

消息体：

```json
{
  "messageId": "支付成功:70001",
  "paymentId": 70001,
  "orderId": 50001,
  "orderNo": "SO202605270001",
  "paidAmount": 9900,
  "paidAt": "2026-05-27 10:05:00"
}
```

消费规则：

- 订单服务更新订单为已支付。
- 库存服务将锁定库存转为已售库存。
- 重复支付成功消息必须幂等处理。

### 10.4 死信消息

死信队列用于承接多次重试仍失败的消息。

建议死信消息包含：

```json
{
  "messageId": "秒杀消息:10001:30001:10001:REQ202605270016",
  "originalExchange": "seckill.order.exchange",
  "originalRoutingKey": "seckill.order.create",
  "payload": {},
  "failureReason": "库存服务调用失败",
  "retryCount": 5,
  "deadAt": "2026-05-27 10:10:00"
}
```

死信处理方式：

- 记录告警。
- 人工查看失败原因。
- 支持修复后重新投递。
- 对无法恢复的消息执行库存回补或订单失败标记。

## 十一、前端调用建议

### 11.1 Axios 封装

前端请求封装建议统一处理：

- 自动附加访问令牌。
- 自动附加请求编号。
- 统一处理未登录。
- 统一处理业务错误码。
- 对秒杀提交按钮做防重复点击。

### 11.2 秒杀提交流程

前端页面建议流程：

1. 进入秒杀详情页，获取服务器时间和活动状态。
2. 活动未开始时展示倒计时。
3. 活动开始后先获取秒杀令牌。
4. 用户点击秒杀按钮后提交请求。
5. 接口返回排队中后禁用按钮。
6. 前端轮询秒杀结果。
7. 成功后跳转订单详情。
8. 失败后展示失败原因。

### 11.3 前端错误提示

| 错误场景 | 前端提示 |
| --- | --- |
| 未登录 | 请先登录后参与秒杀 |
| 活动未开始 | 活动尚未开始 |
| 活动已结束 | 活动已结束 |
| 商品售罄 | 商品已抢光 |
| 重复参与 | 你已经参与过本次秒杀 |
| 排队中 | 请求处理中，请稍候 |
| 系统繁忙 | 当前参与人数较多，请稍后重试 |

## 十二、接口验收标准

- 所有接口返回统一响应格式。
- 所有写接口支持请求编号。
- 登录接口能正确签发令牌。
- 网关能正确拦截未登录请求。
- 商品列表和详情能够正常展示。
- 秒杀提交接口能返回排队中或明确失败原因。
- 同一用户重复提交秒杀不会重复创建订单。
- 秒杀结果接口能查询到最终状态。
- 订单详情接口能查询到秒杀订单。
- RabbitMQ 重复投递时订单服务不会重复处理。
- 消息发送失败后本地消息表能够补偿重投。
