# SnapShop 在线购物秒杀平台数据库设计文档

## 一、文档说明

本文档定义 SnapShop 项目建议使用的 MySQL 核心表结构。字段以业务设计为主，实际开发时可根据 ORM 框架和命名规范微调。

数据库设计目标：

- 支撑用户、商品、库存、秒杀活动、订单等核心业务。
- 支撑秒杀场景下的订单幂等和库存一致性。
- 支撑 RabbitMQ 可靠消息投递和重复消费防护。
- 支撑后续对账、补偿和问题排查。

## 二、命名规范

- 表名使用小写英文和下划线。
- 主键统一使用 `id`。
- 金额统一使用分作为单位，字段类型使用整数。
- 时间字段统一使用 `created_at`、`updated_at`。
- 状态字段统一使用 **`varchar(32)`** 存储英文状态码（如 `PENDING_PAY`、`ON_SALE`），禁止混用小整数与字符串。
- 关键写操作表建议保留 `version` 字段用于乐观锁。

| --- | --- | --- |

| `sku_id` | bigint | 商品规格编号 |

| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

```sql

```

库存扣减建议使用条件更新：

```sql
UPDATE sku_stock
SET available_stock = available_stock - 1,
    locked_stock = locked_stock + 1,
    version = version + 1
WHERE sku_id = ?
  AND available_stock > 0
  AND version = ?;
```

## 八、消息可靠性相关表

### 8.1 生产端本地消息表

表名：`local_message`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `message_id` | varchar(128) | 消息编号 |
| `exchange_name` | varchar(128) | 交换机 |
| `routing_key` | varchar(128) | 路由键 |
| `payload` | text | 消息内容 |
| `status` | varchar(32) | 消息状态 |
| `retry_count` | int | 重试次数 |
| `next_retry_time` | datetime | 下次重试时间 |
| `error_message` | varchar(512) | 错误信息 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

建议索引：

```sql
UNIQUE KEY uk_message_id (message_id);
KEY idx_status_retry_time (status, next_retry_time);
```

状态建议：

| 状态 | 说明 |
| --- | --- |
| `待发送` | 已写入本地消息表，尚未确认发送成功 |
| `已发送` | RabbitMQ 发布确认成功 |
| `发送失败` | 发送失败，等待补偿重投 |
| `已取消` | 业务已取消，不再发送 |

状态建议：

| `死信` | 已进入死信处理 |

## 十、关键约束总结

必须优先落地的唯一约束：

```sql
UNIQUE KEY uk_username (username);
UNIQUE KEY uk_phone (phone);
UNIQUE KEY uk_sku_id (sku_id);
UNIQUE KEY uk_activity_sku (activity_id, sku_id);
UNIQUE KEY uk_user_activity_sku (user_id, activity_id, sku_id);
UNIQUE KEY uk_order_no (order_no);
UNIQUE KEY uk_message_id (message_id);
UNIQUE KEY uk_business_key (business_key);
```

这些约束是防止重复注册、重复库存记录、重复秒杀、重复订单、重复消息处理的重要基础。
