# SnapShop 在线购物秒杀平台 RabbitMQ 可靠消息设计文档

## 一、文档说明

本文档专门描述 SnapShop 项目中 RabbitMQ 的使用方式和可靠性设计，重点解决两个问题：

- 如何防止消息丢失。
- 如何防止重复消费。

秒杀系统不应假设消息只会投递一次。系统设计必须接受 RabbitMQ 可能重复投递消息，并通过业务幂等保证最终结果正确。

## 二、消息使用场景

| 场景 | 生产者 | 消费者 | 说明 |
| --- | --- | --- | --- |
| 秒杀异步下单 | 秒杀服务 | 订单服务 | 削峰填谷，异步创建订单 |
| 订单超时关闭 | 订单服务 | 订单服务 | 订单超时未支付后关闭 |
| 支付成功通知 | 支付服务 | 订单服务、库存服务 | 更新订单状态和确认销量 |
| 库存补偿 | 订单服务 | 库存服务 | 下单失败或订单关闭后回补库存 |
| 死信处理 | RabbitMQ | 补偿任务或人工处理 | 多次失败后进入异常流程 |

## 三、交换机和队列规划

### 3.1 秒杀下单

```text
交换机：seckill.order.exchange
队列：seckill.order.queue
路由键：seckill.order.create
死信交换机：seckill.order.dead.exchange
死信队列：seckill.order.dead.queue
死信路由键：seckill.order.dead
```

### 3.2 订单超时

```text
交换机：order.timeout.exchange
队列：order.timeout.queue
路由键：order.timeout.close
```

### 3.3 支付事件

```text
交换机：payment.event.exchange
队列：payment.success.queue
路由键：payment.success
```

### 3.4 库存补偿

```text
交换机：inventory.compensate.exchange
队列：inventory.compensate.queue
路由键：inventory.compensate.recover
```

## 四、消息结构规范

所有业务消息都必须包含以下公共字段：

| 字段 | 说明 |
| --- | --- |
| `messageId` | 消息唯一编号 |
| `requestId` | 请求编号 |
| `businessKey` | 业务幂等键 |
| `eventType` | 事件类型 |
| `createdAt` | 消息创建时间 |
| `payload` | 业务内容 |

秒杀下单消息示例：

```json
{
  "messageId": "秒杀消息:10001:30001:10001:REQ202605270001",
  "requestId": "REQ202605270001",
  "businessKey": "10001:10001:30001",
  "eventType": "秒杀下单",
  "createdAt": "2026-05-27 10:00:00",
  "payload": {
    "userId": 10001,
    "activityId": 10001,
    "skuId": 30001,
    "quantity": 1,
    "seckillPrice": 9900
  }
}
```

## 五、防止消息丢失设计

### 5.1 生产端本地消息表

秒杀服务在发送 RabbitMQ 消息前，先将消息写入本地消息表。

推荐流程：

```text
Redis 预扣库存成功
-> 写入本地消息表，状态为待发送
-> 发送 RabbitMQ 消息
-> 收到发布确认
-> 更新本地消息表状态为已发送
```

如果发送失败或长时间没有收到确认，则由补偿任务重新发送。

### 5.2 发布确认

生产者必须开启发布确认。发布确认用于确认消息是否到达交换机。

处理策略：

| 确认结果 | 处理方式 |
| --- | --- |
| 成功 | 更新本地消息状态为已发送 |
| 失败 | 更新本地消息状态为发送失败，等待补偿 |
| 超时 | 保持待发送或标记发送失败，等待补偿 |

### 5.3 退回通知

生产者必须开启退回通知。退回通知用于发现消息到达交换机，但没有路由到任何队列。

出现退回通常说明：

- 路由键配置错误。
- 队列绑定缺失。
- 交换机和队列配置不一致。

退回消息不能视为发送成功，需要记录错误并等待修复后重投。

### 5.4 队列和消息持久化

RabbitMQ 侧需要保证：

- 交换机持久化。
- 队列持久化。
- 消息持久化。
- 关键队列配置死信交换机。
- 本地开发使用经典队列；生产环境使用**仲裁队列**（Quorum Queue），配置项 `snapshop.rabbitmq.queue.type=quorum`。

### 5.5 消费端手动确认

消费者必须使用手动确认。

处理策略：

| 业务处理结果 | 消息确认方式 |
| --- | --- |
| 业务成功 | 确认消息 |
| 已经处理过 | 确认消息 |
| 可重试异常 | 拒绝消息并允许重试 |
| 不可恢复异常 | 拒绝消息并进入死信队列 |

不能在刚收到消息时就确认，否则消费者宕机会导致消息丢失。

## 六、防止重复消费设计

### 6.1 重复消费来源

重复消费可能来自：

- 消费者业务处理成功，但确认消息失败。
- 消费者处理超时，RabbitMQ 重新投递。
- 网络抖动导致确认包丢失。
- 人工或补偿任务重新投递消息。
- RabbitMQ 连接断开后恢复。

### 6.2 消息编号幂等

消费端收到消息后，先根据 `messageId` 和消费者组查询消费日志表。

处理规则：

```text
如果消费状态为成功
  -> 直接确认消息
如果没有消费记录
  -> 插入处理中记录
如果消费状态为处理中且超时
  -> 根据业务结果决定继续处理或标记失败
如果消费状态为失败
  -> 允许重试处理
```

### 6.3 业务键幂等

仅依赖 `messageId` 不够，因为不同消息可能表达同一个业务操作。秒杀下单必须使用业务键：

```text
用户编号:活动编号:商品规格编号
```

订单创建前必须检查该业务键是否已经成功创建订单。

### 6.4 数据库唯一约束

秒杀订单表必须增加：

```sql
UNIQUE KEY uk_user_activity_sku (user_id, activity_id, sku_id);
```

这是防重复下单的最后防线。

### 6.5 消费事务边界

订单服务消费秒杀消息时建议事务边界如下：

```text
开启数据库事务
  查询消费日志
  判断消息是否已处理
  写入或更新消费日志为处理中
  检查秒杀订单唯一约束
  调用或执行库存扣减
  创建订单
  创建秒杀订单关系记录
  更新消费日志为成功
提交数据库事务
确认 RabbitMQ 消息
```

如果事务失败，不确认消息。

## 七、失败重试设计

### 7.1 可重试异常

以下异常可以重试：

- 数据库临时连接失败。
- 库存服务短暂不可用。
- 网络超时。
- 死锁或锁等待超时。

### 7.2 不可重试异常

以下异常不建议无限重试：

- 参数格式错误。
- 商品不存在。
- 活动不存在。
- 用户不存在。
- 业务状态已经不允许处理。

不可重试异常应记录失败状态，并根据业务需要回补库存。

### 7.3 重试次数（已锁定）

| 配置项 | Nacos 键 | 默认值 |
| --- | --- | --- |
| 消费最大重试次数 | `snapshop.rabbitmq.consumer.max-retries` | **5** |
| 第 1 次重试延迟 | `snapshop.rabbitmq.consumer.retry-delay-ms[0]` | 0（立即） |
| 第 2 次 | `retry-delay-ms[1]` | 10000 |
| 第 3 次 | `retry-delay-ms[2]` | 60000 |
| 第 4 次 | `retry-delay-ms[3]` | 300000 |
| 第 5 次失败后 | — | 进入死信队列 |

重试策略：

```text
第 1 次失败：立即重试
第 2 次失败：延迟 10 秒
第 3 次失败：延迟 1 分钟
第 4 次失败：延迟 5 分钟
第 5 次失败：进入死信队列
```

## 八、死信队列设计

进入死信队列的原因：

- 消息被拒绝且不重新入队。
- 消息超过最大重试次数。
- 消息过期。
- 队列达到长度限制。

死信处理流程：

```text
消费死信消息
-> 记录死信原因
-> 查询业务当前状态
-> 如果订单已成功，标记消息已处理
-> 如果订单未创建，回补 Redis 库存
-> 如果数据库库存已扣减，执行库存回补
-> 记录人工处理结果
```

死信消息必须可查询、可告警、可人工重投。

## 九、本地消息补偿任务

补偿任务扫描本地消息表：

筛选条件：

```text
状态为待发送或发送失败
并且下次重试时间小于当前时间
并且重试次数未超过上限
```

处理流程：

```text
查询待补偿消息
-> 重新发送 RabbitMQ
-> 发送成功后等待发布确认
-> 更新状态和重试次数
-> 超过上限后告警
```

补偿任务可能导致重复投递，所以消费端幂等必须先完成。

### 9.1 补偿任务配置（已锁定）

| 配置项 | Nacos 键 | 默认值 |
| --- | --- | --- |
| 扫描间隔 | `snapshop.message.compensate.interval-seconds` | **30** |
| 单条最大重试 | `snapshop.message.compensate.max-retries` | **10** |
| 超过上限 | — | 标记失败并告警（见可观测性文档） |

## 十、库存补偿联动

以下情况需要考虑库存补偿：

| 场景 | 处理 |
| --- | --- |
| Redis 预扣成功但消息最终发送失败 | 回补 Redis 秒杀库存 |
| 消息进入死信且订单未创建 | 回补 Redis 秒杀库存 |
| 数据库库存扣减成功但订单创建失败 | 回补 MySQL 库存 |
| 订单超时未支付 | 回补 MySQL 库存，必要时同步 Redis |
| 用户取消待支付订单 | 回补 MySQL 库存，必要时同步 Redis |

库存补偿必须使用业务键保证幂等，避免重复回补。

## 十一、日志字段

所有消息生产和消费日志建议包含：

- `messageId`
- `requestId`
- `businessKey`
- `exchange`
- `routingKey`
- `queue`
- `consumerGroup`
- `retryCount`
- `userId`
- `activityId`
- `skuId`
- `orderNo`
- `errorMessage`

## 十二、验收标准

RabbitMQ 可靠消息设计完成后，应满足：

- 秒杀服务发送失败时，本地消息表有记录。
- 发布确认成功后，本地消息状态变为已发送。
- 路由失败时可以收到退回通知。
- 消费者业务成功后才确认消息。
- 同一消息重复投递不会重复创建订单。
- 同一用户同一活动商品不会重复下单。
- 消息多次失败后可以进入死信队列。
- 死信消息可以触发库存补偿或人工处理。
