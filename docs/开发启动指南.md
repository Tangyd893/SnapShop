# SnapShop 在线购物秒杀平台开发启动指南

## 一、文档目标

本文档用于指导项目从零开始动工。它不是架构说明，而是一份开工顺序清单，帮助开发者知道先做什么、做到什么程度算完成、下一步接什么。

本项目按 **[完整增强版范围说明.md](完整增强版范围说明.md)** 交付。动工前须阅读 **[工程配置说明.md](工程配置说明.md)** 并执行 **[文档一致性检查清单.md](文档一致性检查清单.md)** 自检。

## 二、开工前准备

### 2.1 本地环境建议

| 环境 | 建议版本 | 用途 |
| --- | --- | --- |
| Java | 21（LTS） | 后端服务开发，版本以 `docs/工程配置说明.md` 为准 |
| Maven | 3.9 及以上 | 后端依赖管理 |
| Node.js | 20 及以上 | Vue3 前端开发 |
| Docker | 最新稳定版 | 启动中间件 |
| MySQL | 8.x | 业务数据库 |
| Redis | 7.x | 秒杀库存和缓存 |
| RabbitMQ | 3.13 及以上 | 消息队列 |
| Nacos | 2.x | 注册发现和配置中心 |

### 2.2 推荐工具

- 后端开发：IntelliJ IDEA。
- 前端开发：Visual Studio Code。
- 接口调试：Apifox、Postman 或 Insomnia。
- 数据库管理：DataGrip、DBeaver 或 Navicat。
- 压测工具：JMeter（脚本目录 `testing/load/`）。

## 三、第一天建议任务

第一天不建议直接写秒杀逻辑，先把项目骨架和中间件跑通。

### 3.1 检查目录结构

确认项目包含：

```text
backend/
frontend/
docker/
docs/
testing/
README.md
```

### 3.2 初始化后端父工程

建议先建立 Maven 父工程，再逐步加入服务模块。

第一批建议创建（九个业务模块，与架构一致）：

- `snapshop-common`
- `snapshop-gateway`
- `snapshop-auth`
- `snapshop-user`
- `snapshop-product`
- `snapshop-seckill`
- `snapshop-order`
- `snapshop-inventory`
- `snapshop-payment`

完整增强版中，`snapshop-payment` 在第二周完成模拟支付接口；第一周天仅需创建空模块并注册到 Nacos。

### 3.3 初始化前端工程

前端建议使用 Vue3 + Vite。

第一批只需要完成：

- 路由。
- Axios 请求封装。
- 登录态存储。
- 商品列表页占位。
- 秒杀详情页占位。
- 订单详情页占位。

### 3.4 准备 Docker 中间件

本地开发建议通过 Docker Compose 管理：

- Nacos。
- RabbitMQ。
- MySQL。
- Redis。

第一天目标是容器能启动，不需要一开始写复杂配置。

## 四、第一周建议目标

第一周建议完成基础业务闭环，不追求秒杀高并发。

### 4.1 后端目标

- 网关服务可启动。
- 服务可注册到 Nacos。
- 认证服务可以登录并返回令牌。
- 商品服务可以查询商品列表和详情。
- 订单服务可以创建普通订单。
- 库存服务可以条件扣减库存。

### 4.2 前端目标

- 登录页可调用登录接口。
- 商品列表页可展示后端数据。
- 商品详情页可展示 SKU 和价格。
- 订单详情页可展示订单状态。

### 4.3 数据库目标

第一周至少准备以下表：

- 用户表。
- 商品表。
- 商品规格表。
- 库存表。
- 订单表。
- 订单明细表。

## 五、第二周建议目标

第二周开始进入秒杀核心链路。

### 5.1 秒杀服务

- 秒杀活动表。
- 秒杀活动商品表。
- 秒杀活动列表接口。
- 秒杀商品详情接口。
- 秒杀库存预热到 Redis。
- 秒杀提交接口。

### 5.2 RabbitMQ

- 定义秒杀下单交换机。
- 定义秒杀下单队列。
- 定义下单消息结构。
- 秒杀服务发送下单消息。
- 订单服务消费下单消息。

### 5.3 前端秒杀页面

- 活动倒计时。
- 秒杀按钮状态。
- 提交秒杀请求。
- 展示排队中状态。
- 查询秒杀结果。

## 六、第三周建议目标

第三周重点做可靠性增强。

### 6.1 防重复消费

- 订单服务增加消息消费日志表。
- 消费消息前先判断消息是否已成功处理。
- 秒杀订单增加唯一约束。
- 库存扣减增加业务幂等键。

### 6.2 防消息丢失

- 秒杀服务增加本地消息表。
- RabbitMQ 开启发布确认。
- RabbitMQ 消息、交换机、队列全部持久化。
- 消息发送失败后支持补偿重投。
- 消费失败后支持重试或进入死信队列。

### 6.3 库存补偿

- 订单创建失败时回补 Redis 库存。
- 订单超时未支付时回补 MySQL 库存。
- 定时任务对比 Redis、MySQL、订单数量。

## 七、最小可用闭环

建议最先完成这个闭环：

```text
用户登录
-> 查看商品
-> 查看秒杀活动
-> 提交秒杀
-> RabbitMQ 异步下单
-> 创建订单
-> 查询秒杀结果
-> 查看订单详情
```

这个闭环跑通后，再考虑管理后台、支付接入、监控告警和压测优化。

## 八、每次开发前检查

每次开始写代码前建议确认：

- 当前任务属于哪个里程碑。
- 是否已有接口设计。
- 是否需要数据库表。
- 是否需要 Redis 键。
- 是否需要 RabbitMQ 消息。
- 是否涉及幂等。
- 是否涉及失败补偿。
- 是否需要前端状态变化。

## 九、每次提交前检查

提交代码前建议确认：

- 服务可以启动。
- 接口返回统一格式。
- 写操作有请求编号。
- 数据库脚本同步更新。
- 文档中的接口路径没有过期。
- 秒杀链路不绕过幂等校验。
- RabbitMQ 消费成功后才确认消息。

## 十、推荐动工路线

推荐从以下顺序开始：

1. 后端父工程和公共模块。
2. Docker 中间件。
3. Nacos 注册发现。
4. 网关服务。
5. 登录接口。
6. 商品列表和详情。
7. 库存条件扣减。
8. 秒杀活动查询。
9. 秒杀提交和 Redis 预扣。
10. RabbitMQ 异步下单。
11. 订单消费和创建。
12. 防重复消费。
13. 防消息丢失。
14. 前端完整联调。
15. 压测和优化。
