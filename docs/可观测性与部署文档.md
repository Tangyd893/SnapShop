# SnapShop 可观测性与部署文档

## 一、文档说明

本文档定义 SnapShop **完整增强版**的日志规范、指标、链路追踪、告警与 CI/CD 部署流程。

---

## 二、日志规范

### 2.1 结构化 JSON 日志

每个微服务输出 JSON 行日志，必选字段：

| 字段 | 说明 |
| --- | --- |
| timestamp | ISO-8601 |
| level | INFO/WARN/ERROR |
| service | 服务名，如 snapshop-seckill |
| traceId | OpenTelemetry TraceId |
| requestId | 来自 `X-Request-Id` |
| userId | 已登录时填写 |
| uri | 请求路径 |
| durationMs | 耗时 |
| code | 业务 code，异常时非 0 |

### 2.2 消息与订单日志

秒杀提交、MQ 发送/消费、订单创建须额外打印：`messageId`、`businessKey`、`activityId`、`skuId`、`orderId`。

## 三、监控栈（已选型）

| 组件 | 用途 | 部署方式 |
| --- | --- | --- |
| **Micrometer** | 应用指标暴露 | 各 Spring Boot 服务内置 |
| **Prometheus** | 指标采集 | Docker Compose profile `monitoring` |
| **Grafana** | 仪表盘 | 端口 3000，导入 `docker/grafana/dashboards/` |
| **Alertmanager** | 告警路由 | 可选，与 Grafana 告警二选一 |

### 3.1 必采指标

| 指标名 | 类型 | 说明 |
| --- | --- | --- |
| `http_server_requests_seconds` | histogram | 各接口延迟 |
| `seckill_submit_total` | counter | 秒杀提交次数 |
| `seckill_submit_success_total` | counter | 预扣成功次数 |
| `rabbitmq_queue_messages` | gauge | 队列堆积（需 exporter） |
| `rabbitmq_dead_letter_total` | counter | 死信数量 |
| `order_create_duration_seconds` | histogram | 订单创建耗时 |

### 3.2 Grafana 仪表盘

至少包含：

- 秒杀 QPS 与 P99 延迟
- 秒杀成功/失败率
- RabbitMQ 队列深度与死信计数
- 订单创建耗时

---

## 四、告警规则

| 告警名 | 条件 | 级别 | 处理 |
| --- | --- | --- | --- |
| SeckillQueueBacklog | `seckill.order.queue` 堆积 > 1000 持续 5min | warning | 扩容消费者 |
| DeadLetterSpike | 死信 5min 增量 > 50 | critical | 管理后台查看并处理 |
| SeckillErrorRateHigh | 5xx 率 > 5% 持续 3min | critical | 查日志与依赖 |
| MySQLSlowQuery | 慢查询 > 2s | warning | 优化 SQL |

告警通知：开发环境可仅 Grafana 页面；生产可接 Webhook（钉钉/Slack 占位配置于 `docker/grafana/provisioning/alerting/`）。

## 五、链路追踪（已选型）

| 项 | 选型 |
| --- | --- |
| 标准 | **OpenTelemetry** Java Agent |
| 导出 | OTLP → **Jaeger**（Compose profile `tracing`，端口 16686 UI） |
| 传播 | W3C TraceContext + Baggage 携带 `requestId` |

### 5.1 埋点范围

- 网关 → 各微服务 HTTP 调用
- OpenFeign 调用
- RabbitMQ 发送与消费（通过 OTel 消息语义）
- Redis 操作（可选）

### 5.2 与 requestId 关系

- `X-Request-Id`：业务幂等与前端轮询关联
- `traceId`：分布式调用链关联
- 日志中两者同时输出

## 六、CI/CD（GitHub Actions）

### 6.1 工作流

文件路径：`.github/workflows/ci.yml`

| 阶段 | 触发 | 步骤 |
| --- | --- | --- |
| build-backend | push / PR | `mvn -B verify`（含单元测试） |
| build-frontend | push / PR | `npm ci && npm run build`（web + admin） |
| integration | PR to main | 启动 Compose，跑 `testing/api` 冒烟 |
| docker-build | tag `v*` | 构建各服务镜像推送 GHCR（可选） |

### 6.2 分支策略

| 分支 | 用途 |
| --- | --- |
| main | 可发布，受保护 |
| develop | 日常集成 |
| feature/* | 功能开发 |

### 6.3 环境

| 环境 | 说明 |
| --- | --- |
| local | Docker Compose，Nacos 命名空间 `snapshop-dev` |
| staging | 可选，与 prod 配置隔离 `snapshop-staging` |
| prod | Nacos `snapshop-prod`，密钥来自环境变量非仓库 |

## 九、压测与可观测联动

压测使用 JMeter（`testing/load/seckill-load.jmx`），压测期间须观察：

- Grafana 秒杀 P99、错误率
- RabbitMQ 队列深度
- Jaeger 慢 Span
- 日志中 `requestId` 抽样追踪

压测 SLA 见 [完整增强版范围说明.md](完整增强版范围说明.md) §四。
