# SnapShop 在线购物秒杀平台项目里程碑文档

## 一、文档说明

本文档用于规划 SnapShop 在线购物秒杀平台的研发阶段、阶段目标、交付内容、验收标准和风险控制。项目采用前后端分离和微服务架构，后端使用 Java、Spring Cloud Alibaba、Nacos、OpenFeign、RabbitMQ，前端使用 Vue3，中间件通过 Docker 管理。

里程碑设计原则：

- 先搭建基础骨架，再实现业务闭环。
- 先保证正确性，再追求高并发性能。
- 秒杀链路先跑通，再补齐可靠性和补偿能力。
- 每个阶段都要有可运行、可验证的交付结果。

## 六、第四阶段：秒杀核心链路

### 6.1 阶段目标

实现秒杀活动从展示、提交、排队、异步下单到结果查询的完整闭环。

### 6.2 主要任务

- 设计秒杀活动表和秒杀活动商品表。
- 实现秒杀活动列表和详情接口。
- 实现秒杀商品详情接口。
- 实现秒杀活动库存预热到 Redis。
- 编写 Redis Lua 脚本完成原子校验和预扣库存。
- 实现用户重复参与校验。
- 实现秒杀令牌接口。
- 实现秒杀提交接口。
- 实现 RabbitMQ 秒杀下单交换机、队列和路由键。
- 实现订单服务消费秒杀下单消息。
- 实现秒杀结果查询接口。

### 6.3 核心链路

```text
用户点击秒杀
-> 前端提交请求
-> 网关鉴权和限流
-> 秒杀服务校验活动和用户
-> Redis 原子预扣库存
-> RabbitMQ 投递下单消息
-> 订单服务消费消息
-> 数据库扣减库存并创建订单
-> 前端查询秒杀结果
```

### 6.4 交付物

| 交付物 | 说明 |
| --- | --- |
| 秒杀服务 | 活动查询、库存预热、秒杀提交 |
| Redis 脚本 | 原子扣减库存和用户标记 |
| RabbitMQ 配置 | 秒杀下单队列和死信队列雏形 |
| 订单消费逻辑 | 消费消息并创建订单 |
| 秒杀结果接口 | 查询排队中、成功、失败等状态 |

### 6.5 验收标准

- 活动未开始时不能提交秒杀。
- 活动结束后不能提交秒杀。
- 库存不足时返回售罄。
- 同一用户重复提交不会重复下单。
- 秒杀提交成功后返回排队中。
- 订单服务能消费消息并创建订单。
- 前端能查询到最终秒杀结果。

### 6.6 风险与应对

| 风险 | 应对 |
| --- | --- |
| 秒杀接口直接压垮数据库 | 秒杀入口只访问 Redis 和 RabbitMQ |
| 用户重复点击造成重复请求 | 前端禁用按钮，后端做幂等校验 |
| Redis 扣减成功但消息发送失败 | 下一阶段引入本地消息表和补偿 |

## 七、第五阶段：可靠性增强

### 7.1 阶段目标

重点解决项目核心技术难点：防止重复消费、防止信息丢失，并完善库存补偿和对账机制。

### 7.2 主要任务

- 引入生产端本地消息表。
- 开启 RabbitMQ 发布确认。
- 开启 RabbitMQ 退回通知。
- 设置交换机、队列和消息持久化。
- 实现本地消息补偿重投任务。
- 引入消费端消息日志表。
- 实现消费端手动确认。
- 实现重复消息快速确认。
- 实现订单创建幂等。
- 为秒杀订单增加数据库唯一约束。
- 增加死信队列处理流程。
- 实现库存回补任务。
- 实现 Redis 库存和 MySQL 库存对账任务。

### 7.3 防重复消费验收点

- 同一条 RabbitMQ 消息重复投递多次，只创建一个订单。
- 消费者处理成功但确认消息失败时，再次消费不会重复扣库存。
- 同一用户对同一活动商品重复提交，只能有一个成功结果。
- 消息消费日志能记录成功、失败、重试状态。

### 7.4 防消息丢失验收点

- 消息发送失败时，本地消息表保留待发送记录。
- 发布确认失败时，补偿任务可以重新投递。
- 队列重启后，持久化消息不丢失。
- 消费失败时，消息可以重试或进入死信队列。
- 死信消息可以被查询、告警和人工处理。

### 7.5 交付物

| 交付物 | 说明 |
| --- | --- |
| 本地消息表 | 记录生产端消息发送状态 |
| 消费日志表 | 记录消费端处理状态 |
| 补偿任务 | 扫描并重投异常消息 |
| 死信队列 | 承接多次失败消息 |
| 库存对账任务 | 校验 Redis 和 MySQL 库存 |
| 幂等约束 | 数据库唯一约束和业务幂等键 |

### 7.6 风险与应对

| 风险 | 应对 |
| --- | --- |
| 补偿任务重复投递消息 | 消费端必须幂等 |
| 死信队列堆积 | 增加监控告警和人工处理流程 |
| 库存补偿重复执行 | 库存流水使用业务键保证幂等 |

### 7.7 当前落地进展

- 订单服务秒杀 MQ 消费入口已统一事务边界，消费日志、库存扣减、订单创建、秒杀结果写 Redis 放在同一处理链路中。
- 消费端已按 `messageId + consumerGroup` 做消息级幂等，并按 `userId + activityId + skuId` 做业务级兜底。
- 订单侧库存扣减和回补 DTO 已补齐 `businessKey`、`scene`、`activityId` 等字段，便于库存服务保持幂等和秒杀库存回补。
- 秒杀本地消息待发送状态已对齐为 `PENDING_SEND`，避免补偿任务扫描不到待发送消息。
- 网关与秒杀、订单、库存四服务均已接入 Sentinel 流量防护（见 Sentinel 集成计划）。

## 十三、关键质量指标

| 指标 | 目标 |
| --- | --- |
| 库存准确性 | 不允许超卖 |
| 订单幂等性 | 不允许同一用户同一秒杀商品重复下单 |
| 消息可靠性 | 发送失败可补偿，消费失败可追踪 |
| 接口一致性 | 返回格式和错误码统一 |
| 可启动性 | 新环境可通过文档完成启动 |
| 可测试性 | 核心链路有接口测试和压测脚本 |

## 十四、项目风险清单

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 微服务数量较多 | 开发和启动成本上升 | 初期按核心服务优先实现，非核心服务可模拟 |
| 秒杀链路复杂 | 容易出现库存和订单不一致 | 先跑通链路，再补可靠性和对账 |
| RabbitMQ 可靠性配置遗漏 | 可能导致消息丢失 | 明确发布确认、持久化、手动确认、死信队列 |
| 重复消费处理不足 | 可能重复创建订单 | 消息编号、业务键、唯一约束三层幂等 |
| 前后端联调字段变化 | 页面开发返工 | 以接口设计文档为准，变更时同步文档 |
| 压测数据不可控 | 测试结果不可信 | 固定初始化数据和压测脚本 |

## 十五、总结

SnapShop 项目应按照“基础环境、微服务骨架、基础业务、秒杀链路、可靠性增强、前端联调、测试优化、部署文档”的顺序推进。项目最重要的验收点不是单纯完成页面或接口，而是在高并发和异常场景下仍能做到不超卖、不重复下单、消息可追踪、失败可补偿。
