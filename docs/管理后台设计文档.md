# SnapShop 管理后台设计文档

## 一、文档说明

本文档定义 SnapShop **完整增强版**管理后台（B 端）的页面、角色权限、API 与任务编号。管理后台与 C 端 `snapshop-web` 分离，建议使用独立前端工程 `frontend/snapshop-admin`（Vue3 + Vite + Element Plus）。

---

## 二、角色与权限（RBAC）

| 角色 | 编码 | 权限范围 |
| --- | --- | --- |
| 超级管理员 | `SUPER_ADMIN` | 全部功能 |
| 运营 | `OPERATOR` | 商品、秒杀活动、订单查询 |
| 客服 | `SUPPORT` | 订单查询、死信查看（只读） |

认证方式：管理员账号密码登录，JWT 与 C 端隔离（不同 `issuer` 与密钥，配置于 `snapshop-admin.yaml`）。

---

## 三、页面清单

| 路由 | 页面 | 角色 |
| --- | --- | --- |
| `/login` | 管理员登录 | 公开 |
| `/dashboard` | 概览（订单量、秒杀成功率、MQ 堆积） | 全部 |
| `/products` | 商品列表 | OPERATOR+ |
| `/products/:id` | 商品编辑 | OPERATOR+ |
| `/categories` | 分类管理 | OPERATOR+ |
| `/seckill/activities` | 秒杀活动列表 | OPERATOR+ |
| `/seckill/activities/:id` | 活动配置与商品绑定 | OPERATOR+ |
| `/seckill/warmup` | 活动预热（写 Redis） | OPERATOR+ |
| `/orders` | 订单查询 | OPERATOR+ |
| `/orders/:id` | 订单详情与状态流水 | OPERATOR+ |
| `/mq/dead-letters` | 死信消息列表与重投 | SUPER_ADMIN |
| `/users` | C 端用户查询（只读） | SUPPORT+ |
| `/admins` | 管理员账号管理 | SUPER_ADMIN |

## 四、管理 API（网关前缀 `/api/admin`）

网关路由到 `snapshop-admin` 服务（端口 8090），或初期合并到各业务服务的 `/internal/admin` 路径（实现时二选一，推荐独立 admin 服务）。

### 4.1 认证

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/admin/auth/login` | 管理员登录 |
| POST | `/api/admin/auth/logout` | 退出 |

### 4.2 商品

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/admin/products` | 分页列表 |
| POST | `/api/admin/products` | 创建商品 |
| PUT | `/api/admin/products/{id}` | 更新商品 |
| PUT | `/api/admin/products/{id}/status` | 上下架，`ON_SALE` / `OFF_SALE` |
| GET | `/api/admin/categories` | 分类列表 |
| POST | `/api/admin/categories` | 创建分类 |

### 4.3 秒杀活动

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/admin/seckill/activities` | 活动列表 |
| POST | `/api/admin/seckill/activities` | 创建活动 |
| PUT | `/api/admin/seckill/activities/{id}` | 更新活动 |
| POST | `/api/admin/seckill/activities/{id}/items` | 绑定秒杀商品 |
| POST | `/api/admin/seckill/activities/{id}/warmup` | 触发 Redis 预热 |

预热逻辑与 [架构设计文档.md](架构设计文档.md) §7.1 一致。

### 4.4 订单与运维

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/admin/orders` | 按状态、用户、时间筛选 |
| GET | `/api/admin/orders/{id}` | 订单详情含状态流水 |
| GET | `/api/admin/mq/dead-letters` | 死信分页 |
| POST | `/api/admin/mq/dead-letters/{id}/requeue` | 人工重投（须幂等） |

### 4.5 统一响应

与 C 端一致：`code`、`message`、`data`、`requestId`、`timestamp`（见接口设计文档 §2.3）。

## 五、数据库扩展

### 5.1 管理员表 `admin_user`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| username | varchar(64) | 登录名，唯一 |
| password_hash | varchar(128) | BCrypt |
| role | varchar(32) | SUPER_ADMIN / OPERATOR / SUPPORT |
| status | varchar(32) | ACTIVE / DISABLED |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 5.2 与现有表关系

- 商品、秒杀、订单表与 C 端共用，管理后台通过 admin 服务调用各业务服务 Feign 或直连 DAO（推荐 Feign 调用 product/seckill/order 服务）。

## 七、验收标准

- 运营可通过后台创建商品并配置秒杀活动。
- 预热后 C 端可看到活动并参与秒杀。
- 客服可查询订单；超级管理员可处理死信。
- 管理 JWT 不能访问 C 端用户接口，反之亦然。
